# 캐시 구조 이해하기 (mc-o11y-manager)

이 문서는 매니저(`java/mc-o11y-manager`)가 화면을 빠르게 보여주려고 어떻게 **캐시(미리 저장해 둔
임시 결과)** 를 쓰는지 설명합니다.

## 캐시가 뭐고 왜 쓰나요?

대시보드를 열면 매니저는 매번 **InfluxDB(메트릭 저장소)** 나 **cb-spider(클라우드에서 사용량을
가져오는 도구)** 에 물어봐서 그래프를 그립니다. 그런데 여러 사람이 같은 화면을 자주 열면, 똑같은
질문을 계속 던지게 되고 그때마다 느려집니다.

그래서 한 번 가져온 답을 잠깐 메모리에 적어두고, 같은 질문이 또 오면 **저장소에 다시 묻지 않고
적어둔 답을 바로 돌려줍니다.** 이게 캐시입니다. 카페에서 자주 나가는 메뉴를 미리 만들어 두는 것과
비슷합니다.

이 프로젝트에서 알아둘 점 세 가지:

- 캐시는 [Caffeine](https://github.com/ben-manes/caffeine)이라는 자바 라이브러리로 **직접** 만들었습니다.
  (스프링의 자동 캐시 기능 `@Cacheable`은 쓰지 않습니다.)
- 캐시는 전부 **매니저 프로그램 한 개의 메모리 안**에만 있습니다. 매니저가 꺼지면 사라집니다.
- 주로 쓰는 캐시는 **두 개**입니다.
  1. **모니터링 메트릭 캐시** — InfluxDB에서 가져온 그래프 데이터를 저장
  2. **CSP 모니터링 캐시** — cb-spider에서 가져온 클라우드 사용량을 저장

그리고 각 캐시에는 **워머(warmer, 미리 데워두는 일꾼)** 가 붙어 있습니다. 사용자가 화면을 열기
전에 미리 자주 보는 VM의 데이터를 가져다 채워 둬서, 처음 열 때부터 빠르게 보이도록 합니다.

```
                          ┌──────────── 매니저 프로그램(메모리 안) ────────────┐
 사용자 / 화면            │                                                     │
 ───────────►  InfluxDBController ─► [모니터링 메트릭 캐시] ──(없으면)──► InfluxDB
 (VM 대시보드)            │            있으면 바로 응답                          │
                          │                                                     │
 ───────────►  CspMonitoringController ─► [CSP 모니터링 캐시] ──(없으면)──► cb-spider ─► 클라우드 API
 (클라우드 사용량)        │            있으면 바로 응답                          │
                          └─────────────────────────────────────────────────────┘
```

> **용어 한 줄 정리**
> - **TTL(유효기간)**: 저장한 값이 살아있는 시간. 지나면 버려집니다.
> - **워밍(warming)**: 사용자가 보기 전에 미리 캐시를 채워 두는 것.
> - **히트(hit) / 미스(miss)**: 물어봤을 때 캐시에 있으면 히트, 없어서 저장소까지 가야 하면 미스.
> - **무효화(invalidate)**: 저장해 둔 값을 전부 지우는 것.

## 1. 모니터링 메트릭 캐시 (InfluxDB 그래프 데이터)

InfluxDB에서 가져온 그래프 데이터를 저장해서, **같은 시간대에 똑같은 조회가 또 오면 InfluxDB에
다시 묻지 않고** 바로 돌려줍니다.

| 항목 | 위치 |
|---|---|
| 캐시 본체 | `service/cache/MonitoringCacheService.java` |
| 저장 이름표(키) | `service/cache/MonitoringCacheKey.java` |
| 설정 | `config/MonitoringCacheProperties.java` (`monitoring.cache.*`) |
| 미리 데우는 일꾼 | `service/cache/MonitoringCacheWarmScheduler.java` |
| 사용하는 곳 | `service/InfluxDbServiceImpl.java`의 메트릭 조회 |
| 관리 API | `controller/InfluxDBController.java` |

### 어떻게 동작하나요

조회 코드는 항상 "있으면 꺼내 쓰고, 없으면 가져와서 저장"하는 `getOrLoad(...)`를 거칩니다.

```java
return monitoringCacheService.getOrLoad(
        nsId, infraId, nodeId, req, () -> loadMetricsByVM(nsId, infraId, nodeId, req));
```

캐시에 없을 때(미스)만 뒤의 함수가 실제로 InfluxDB를 조회하고, 그 결과를 저장한 뒤 돌려줍니다.

### 같은 조회인지 어떻게 구분하나요 (저장 이름표)

저장할 때 붙이는 이름표는 `(어느 네임스페이스 / 어느 인프라 / 어느 노드 / 어떤 조회조건 / 몇 시
구간)` 으로 만들어집니다. 두 가지가 포인트입니다.

- **"몇 시 구간"** : 시간을 1시간 단위로 끊어서 같은 칸에 넣습니다. 그래서 **같은 1시간 안에 들어온
  똑같은 조회는 같은 칸을 함께 쓰고**, 시간이 바뀌면 새 칸(새 데이터)이 만들어집니다. 오래된 값을
  계속 보여주지 않으면서도 재사용할 수 있게 해 주는 장치입니다.
- **"어떤 조회조건"** : measurement, range, group_time, limit, group_by, 함수/필드 같은 조건을
  순서에 상관없이 똑같은 문자열로 만듭니다. (ns/infra/node는 이미 이름표에 따로 들어가므로 여기선 뺍니다.)

### 언제 지워지나요 (용량과 유효기간)

- **개수가 아니라 용량(메모리 크기)으로 관리**합니다. 전체 합이 기본 **512MB**를 넘으면, 오래 안
  쓰인 것부터 자동으로 버립니다. 즉 자주 보는 VM 데이터가 자연스럽게 남습니다.
- **유효기간은 VM이 만들어진 시각을 기준**으로 정합니다.
  - 보통은 `VM 생성시각 + 7일`까지 살아있습니다 (최대 7일).
  - VM이 **이미 만들어진 지 7일이 넘었으면 아예 저장하지 않습니다.** 어차피 곧 만료될 거라
    자리만 차지하기 때문입니다.
  - 값을 **읽는다고 수명이 늘어나지는 않습니다.** 처음 정해진 만료시각 그대로입니다.
- **빈 결과는 저장해도 "없는 것"으로 칩니다.** 데이터가 0개인 응답이 들어오면 히트로 안 쳐서,
  나중에 데이터가 실제로 쌓이기 시작하면 다음 조회에서 진짜 값으로 채워집니다.

### 미리 데워두기 (워머)

일꾼은 먼저 **지금 데이터가 있는 VM들**을 모아 **최근에 만들어진 순서로 정렬**한 뒤, 상위 몇 개
(기본 **10개**)만 골라 미리 조회해서 캐시에 채워 둡니다. 세 종류가 돌아갑니다.

| 작업 | 주기(기본) | 무엇을 데우나 | 비고 |
|---|---|---|---|
| `realtime` | 매 1분 | 최근 1시간 / 6시간 / 12시간 그래프 | 거의 실시간으로 유지 |
| `longrange` | 매시 5분 | 1일 / 3일 / 5일 / 7일 그래프 | 장기 데이터는 다운샘플링 DB에서 가져옴 |
| `overview` | 매 1분 | NS/MCI 개요 화면이 쓰는 cpu·mem·disk 그래프 | 개요 화면과 **똑같은 형태**로 데워야 화면 열 때 캐시가 맞아떨어짐 |

> 왜 `overview`가 따로 있나요? `realtime`이 데워 두는 형태와 개요 화면이 실제로 보내는 조회 형태가
> 조금 달라서, 그대로 두면 매번 미스가 납니다. 그래서 화면이 쓰는 형태 그대로 미리 데워 둡니다.

### 관리 API (`InfluxDBController`)

| 방법 | 경로 | 하는 일 |
|---|---|---|
| `GET` | `/api/o11y/monitoring/influxdb/cache/stats` | 적중률, 저장 개수, 버려진 수 등 통계 보기 |
| `DELETE` | `/api/o11y/monitoring/influxdb/cache` | 캐시 전부 비우기 |
| `POST` | `/api/o11y/monitoring/influxdb/cache/warm` | 지금 당장 미리 데우기 실행 |

## 2. CSP 모니터링 캐시 (cb-spider 클라우드 사용량)

cb-spider에서 가져온 클라우드 사용량 응답을 저장합니다. NS/MCI 개요 화면은 탭을 한 번 열 때마다
**약 8개 지표 × VM 여러 대** 만큼 조회를 쏟아내고, 그때마다 cb-spider가 클라우드 API까지 다녀오기
때문에 느립니다. 이 반복 조회를 짧은 시간 동안 캐시가 대신 받아 줍니다.

| 항목 | 위치 |
|---|---|
| 캐시 본체 | `service/cache/CspCacheService.java` |
| 저장 이름표(키) | `service/cache/CspCacheKey.java` |
| 설정 | `config/CspCacheProperties.java` (`csp.cache.*`) |
| 미리 데우는 일꾼 | `service/cache/CspCacheWarmScheduler.java` |
| 사용하는 곳 | `controller/CspMonitoringController.java` |

### 같은 조회인지 어떻게 구분하나요

이름표는 `(VM인지 클러스터노드인지 / 대상 이름 / 지표 / 커넥션 / 몇 시간 전부터 / 몇 분 간격)`
으로 만듭니다.

### 언제 지워지나요

- **개수로 관리**합니다. 최대 **1만 개**까지 저장하고, 넘으면 오래된 것부터 버립니다.
- **유효기간은 5분**입니다. 워밍 주기(1분/5분)보다 길어서 갱신 사이에도 값이 유효합니다.
- **빈 응답은 저장하지 않습니다.** 잠깐 비어서 온 응답이 멀쩡히 들어있던 값을 덮어쓰지 않게 합니다.

### 미리 데워두기 (워머)

일꾼 스레드 기본 **30개**, 네임스페이스마다 최근 VM **20개**까지 데웁니다. `aws`, `azure`, `gcp`
커넥션만 대상입니다. 그리고 "어떤 커넥션·클러스터가 있는지" 목록 조회는 Tumblebug가 과부하로
막히지 않도록 **5분 동안 기억해 두고 재사용**합니다.

| 작업 | 주기(기본) | 무엇을 데우나 (몇 시간 전 / 몇 분 간격) |
|---|---|---|
| `realtime` | 매 1분 | 1시간 / 6시간 / 12시간 전 (5분 간격) |
| `longrange` | 매 5분 | 1일 / 3일 / 5일 / 7일 전 (5분 간격) |

### 관리 API (`CspMonitoringController`)

| 방법 | 경로 | 하는 일 |
|---|---|---|
| `GET` | `…/cache/stats` | 통계 보기 |
| `DELETE` | `…/cache` | 캐시 전부 비우기 |

(이 캐시에는 "지금 데우기" HTTP 버튼은 없고, 정해진 주기로만 데웁니다.)

## 3. 뒤에서 돕는 작은 캐시들

### VM 생성시각 기억 (VmCreatedTimeResolver)
`service/cache/VmCreatedTimeResolver.java` — VM이 언제 만들어졌는지를 CB-Tumblebug에서 한 번
물어보고 **1시간 동안 기억**합니다(최대 1만 개). 매번 Tumblebug에 묻지 않으려는 것이며, "생성시각을
못 찾았다"는 결과조차 기억해서 헛질문을 줄입니다. 위의 메트릭 캐시가 유효기간을 계산할 때, 그리고
워머가 "최근 VM"을 고를 때 씁니다.

### 토폴로지 목록 기억 (CSP 워머 내부)
커넥션 목록과 클러스터 목록을 **5분간 기억**해서, 워밍이 돌 때마다 매번 다시 조회하지 않습니다.

### SSH 연결 재사용 (SshServiceImpl)
대상 서버로 맺은 SSH 연결을 잠깐 보관해 두고 재사용합니다. **120초** 동안 안 쓰이면 끊습니다.
메트릭 캐시는 아니지만, 같은 "재사용으로 빨라지게 하는" 장치라 함께 적어 둡니다.

## 4. 설정값 한눈에 보기

모든 값은 환경변수로 바꿀 수 있습니다 (기본값은 `src/main/resources/application.yaml`).

### 모니터링 메트릭 캐시 (`monitoring.cache.*`)

| 환경변수 | 기본값 | 의미 |
|---|---|---|
| `MONITORING_CACHE_ENABLED` | `true` | 이 캐시를 켤지 |
| `MONITORING_CACHE_BLOCK_PERIOD_SECONDS` | `3600` | 시간을 몇 초 단위로 끊을지 (1시간) |
| `MONITORING_CACHE_MAX_WEIGHT_MB` | `512` | 캐시 최대 메모리 크기 |
| `MONITORING_CACHE_EXPIRE_SECONDS` | `604800` | 최대 유효기간 (7일) |
| `MONITORING_CACHE_BYTES_PER_POINT` | `200` | 데이터 한 점이 대략 몇 바이트인지(크기 계산용) |
| `MONITORING_CACHE_WARM_ENABLED` | `true` | 미리 데우기 켤지 |
| `MONITORING_CACHE_WARM_TOP_N` | `10` | 한 번에 데울 최신 VM 수 |
| `MONITORING_CACHE_WARM_REALTIME_CRON` / `_THREADS` | `0 * * * * *` / `10` | realtime 작업 주기 / 스레드 |
| `MONITORING_CACHE_WARM_LONGRANGE_CRON` / `_THREADS` | `0 5 * * * *` / `10` | longrange 작업 주기 / 스레드 |
| `MONITORING_CACHE_WARM_OVERVIEW_ENABLED` / `_CRON` / `_THREADS` | `true` / `0 * * * * *` / `10` | overview 작업 |

### CSP 모니터링 캐시 (`csp.cache.*`)

| 환경변수 | 기본값 | 의미 |
|---|---|---|
| `CSP_CACHE_ENABLED` | `true` | 이 캐시를 켤지 |
| `CSP_CACHE_MAX_SIZE` | `10000` | 최대 저장 개수 |
| `CSP_CACHE_EXPIRE_SECONDS` | `300` | 유효기간 (5분) |
| `CSP_CACHE_WARM_ENABLED` | `true` | 미리 데우기 켤지 |
| `CSP_CACHE_WARM_THREADS` | `30` | 워머 스레드 수 |
| `CSP_CACHE_WARM_TOP_N` | `20` | 네임스페이스당 데울 최신 VM 수 |
| `CSP_CACHE_WARM_REALTIME_CRON` | `0 * * * * *` | realtime 작업 주기 |
| `CSP_CACHE_WARM_LONGRANGE_CRON` | `0 */5 * * * *` | longrange 작업 주기 |

## 5. 매니저를 여러 대로 늘리면?

캐시는 전부 매니저 **한 대의 메모리 안에만** 있습니다. 매니저를 두 대 이상으로 늘리면:

- 캐시가 **서로 공유되지 않습니다.** 각 매니저가 따로 데우고 따로 답합니다.
- "캐시 비우기"를 호출해도 **그 요청을 받은 매니저만** 비워집니다.
- 적중률 통계도 매니저마다 따로입니다.

지금처럼 매니저 한 대로 쓰는 기본 구성에서는 문제가 없습니다. 매니저를 여러 대로 늘릴 때에만
공유 캐시(예: Redis 같은 외부 캐시)가 필요해집니다.

## 6. 왜 이렇게 만들었나 (설계 의도)

- **왜 스프링 자동 캐시(`@Cacheable`)를 안 썼나?** 자동 기능으로는 표현하기 어려운 동작이 필요해서
  직접 만들었습니다: 용량 기준으로 버리기, VM 생성시각에 맞춘 유효기간, "빈 결과는 없는 셈 치기",
  미리 데워 두기 같은 것들입니다.
- **왜 "빈 결과는 없는 셈" 치나?** 메트릭은 VM이 켜진 직후엔 아직 안 쌓여 있고, 워머가 데이터가
  들어오기 전 타이밍에 돌 수도 있습니다. 빈 응답을 저장해 버리면 유효기간 동안 계속 "데이터 없음"만
  보여 주게 되므로, 빈 건 저장하지 않고 다음에 다시 시도합니다.
- **왜 최신 VM부터 데우나?** 방금 만든 VM을 사람들이 가장 많이 들여다보고, 그 짧은 구간은 캐시로
  들고 있기에도 가볍기 때문입니다. 오래된 VM은 필요할 때 그때그때 조회합니다.
